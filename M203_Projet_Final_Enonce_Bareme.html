Projet final — collecter, pour de vrai

NEEKOCODE · Formation IPSSI · lancement J4 après-midi · soutenances J5 après-midi
L'esprit. Toute la semaine, on a appris à décider : quel barreau de l'échelle, quel outil, quel risque. Le projet n'est pas un exercice de plus — c'est l'épreuve où tu prouves que tu sais choisir un régime et le défendre. On ne notera pas « ça tourne ». On notera les décisions que tu assumes et la résilience de ta collecte.

Cadre

  • Groupes : binôme ou trinôme.
  • Sujets : 2 sujets imposés (ci-dessous) ou 1 sujet libre, à me faire valider en recon clinic.
  • Cibles reproductibles uniquement : aucune cible protégée par un mur anti-bot (jamais de note qui dépend d'un blocage).
  • Calendrier : recon clinic + build (J4 PM) → build en autonomie (J5 matin) → soutenances (J5 PM, ~15 min + ~10 min Q&A).
  • LLM : bonus, jamais obligatoire. Outils no-code : interdits comme livrable.

Sujet 1 — Veille citations & auteurs

Imposé · décision API / rendu JS

Cible : quotes.toscrape.com et ses variantes (/ statique, /js rendu JavaScript, /scroll scroll infini, /search en POST).

Mission : collecter toutes les citations (texte, auteur, tags) + la bio de chaque auteur (page détail), stocker en base, de façon résiliente et idempotente.

Le sel du sujet

  • Tu dois choisir par quelle porte entrer et le défendre : réflexe API cachée derrière /scroll (barreau 1) ? Playwright sur /js (barreau 3) ? httpx+BS4 sur le statique (barreau 2) ?
  • Pas de Scrapy attendu ici : c'est un sujet de décision et de rendu.
  • Bonus LLM : catégoriser/tagger automatiquement les citations par thème.

Sujet 2 — Collecte industrielle d'un palmarès sportif

Imposé · Scrapy & résilience

Cible : scrapethissite.com/pages/forms/ (Hockey Teams) — ~580 équipes sur ~24 pages paginées, avec un formulaire de recherche/filtre.

Mission : un projet Scrapy propre (spider + items + pipeline + settings) qui collecte tout le palmarès (équipe, année, victoires, défaites, % de victoires…), le stocke en base, et tient en production : politesse, retries, idempotence (relançable chaque nuit sans doublon).

Le sel du sujet

  • Volume + pagination + robustesse = le terrain où Scrapy se justifie (les deux critères vus en J3).
  • Exploiter le formulaire pour filtrer (ex. par saison).
  • Bonus : la section /pages/ajax-javascript/ (Oscars) est en AJAX → applique le réflexe API cachée dessus.
  • Bonus LLM : normaliser/enrichir des champs hétérogènes.

Sujet 3 — Libre (à valider)

Au choix · validé en recon clinic

Ta propre cible, sous réserve de validation par moi via 3 questions go/no-go :

  • Régime identifiable ? Sais-tu dire quel barreau elle exige (et pourquoi) ?
  • Reproductible ? La cible répond pour tout le monde, sans mur anti-bot ?
  • Cadrée ? Le périmètre est faisable en ~1,5 jour, sans dérive ?

Contraintes techniques (communes)

  • Environnement géré avec uv ; dépendances déclarées (requirements.txt ou pyproject.toml).
  • Stockage en base (SQLite minimum) avec idempotence : relancer la collecte ne crée pas de doublons.
  • Résilience : politesse (délai/UA), retries sur erreurs transitoires, gestion d'erreurs propre.
  • Structure du code :
    • Sujets 1 & libre (script) : app.py (point d'entrée) + utiles.py (fetch / parse / store réutilisables) + un module BDD.
    • Sujet 2 (Scrapy) : structure de projet Scrapy (comme le TP3) — spider, items, pipelines, settings.
  • Repo GitHub : commits réguliers, au moins 1 commit par membre.
  • README qui DÉFEND les décisions (le cœur de la note) : quel barreau et pourquoi pas celui du dessus, ce que la recon a montré, choix de résilience, lecture du robots.txt, remarque RGPD, alternatives écartées.
  • Si LLM utilisé : tool use + validation de la sortie (jamais en aveugle).

Recon clinic (J4 après-midi) — le passage obligé

Aucune ligne de code avant ce feu vert. Chaque groupe vient me présenter SA cible et son régime. C'est le moment qui sauve les projets : on évite de coder 3 h sur la mauvaise approche.
  • DevTools → Network : y a-t-il une API qui sert la donnée ? (barreau 1)
  • view-source : la donnée est-elle dans le HTML d'origine ? (statique → barreau 2)
  • Test de blocage : la cible répond-elle proprement à une requête polie ? (pas de mur)
  • Sujet libre : les 3 questions go/no-go ci-dessus.

Soutenance (J5 après-midi) — ~15 min + ~10 min Q&A

Attendu de la présentation

  • La cible et le régime choisi — et surtout pourquoi (quel barreau, pourquoi pas celui du dessus).
  • Une démo qui tourne (même partielle).
  • La résilience démontrée (relance idempotente, comportement face à une erreur).
  • Limites connues + ce que vous feriez avec une semaine de plus.

Questions types du jury (pour vous préparer)

  • « Et si le site changeait de structure demain — votre collecte tient ou casse ? »
  • « Pourquoi ce barreau et pas celui du dessus / du dessous ? »
  • « Comment savez-vous que vous n'avez pas de doublons en base ? »
  • « Qu'avez-vous lu dans le robots.txt ? Une question RGPD sur ces données ? »
